# Feature Specification: 圈子关注功能

**Feature Branch**: `001-circle-follow`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "我想针对圈子中实现关注功能的开发，针对发圈的用户可以进行关注以及取消关注操作，关注的人在关注标签的下方一行展示"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 关注发圈用户 (Priority: P1)

用户在浏览圈子动态时，可以在每条动态的作者信息区域看到"关注"按钮，点击后即可关注该发圈用户。若当前用户已关注该作者，则显示"已关注"，可点击取消关注。自己发布的动态不显示关注按钮。

**Why this priority**: 关注操作是本功能的核心交互，没有关注能力，后续的关注列表展示和关注流筛选均无意义。

**Independent Test**: 打开圈子列表，找到他人动态，点击"关注"按钮，按钮状态变为"已关注"，再次点击变回"关注"，即可验证核心关注/取消关注交互完整可用。

**Acceptance Scenarios**:

1. **Given** 用户浏览圈子动态列表，**When** 该动态的作者不是当前登录用户，**Then** 动态作者区域显示"关注"按钮（若已关注则显示"已关注"）
2. **Given** 用户点击某动态的"关注"按钮，**When** 关注操作成功，**Then** 按钮状态即时切换为"已关注"，后端持久化该关注关系
3. **Given** 用户点击"已关注"按钮，**When** 取消关注操作成功，**Then** 按钮状态切换回"关注"，后端删除该关注关系
4. **Given** 动态的作者是当前登录用户自己，**When** 浏览该动态，**Then** 不显示关注/已关注按钮
5. **Given** 网络异常导致关注/取消关注失败，**When** 操作响应失败，**Then** 按钮状态回退至操作前，并给出简短错误提示

---

### User Story 2 - 关注标签下展示关注用户列表 (Priority: P2)

当用户切换到"关注"筛选标签时，在筛选标签行的正下方一行横向展示当前用户已关注的所有用户头像及姓名缩略信息，以便快速识别关注对象。该区域在其他筛选标签激活时不显示。

**Why this priority**: 关注用户列表横向展示是本需求明确要求的 UI，帮助用户直观看到自己关注了哪些人，并为后续跳转用户主页等操作奠定基础。

**Independent Test**: 切换至"关注"标签，关注用户一行出现在标签行正下方，横向排列已关注用户的头像与姓名，可独立验证展示逻辑与空状态处理。

**Acceptance Scenarios**:

1. **Given** 用户切换到"关注"标签，**When** 已有关注用户，**Then** 筛选标签行正下方展示一行横向可滚动的已关注用户列表（头像 + 姓名）
2. **Given** 用户切换到"关注"标签，**When** 尚未关注任何人，**Then** 该行显示空状态提示，如"暂无关注，去圈子发现感兴趣的人吧"
3. **Given** 关注用户数量超过屏幕宽度，**When** 用户水平滑动该行，**Then** 可平滑横向滚动查看所有关注用户
4. **Given** 用户切换到"关注"以外的其他筛选标签，**When** 标签切换完成，**Then** 关注用户列表行隐藏，不占用任何布局空间
5. **Given** 用户在"关注"标签下执行了取消关注操作，**When** 操作成功，**Then** 关注用户列表行即时移除该用户的条目

---

### User Story 3 - 关注流动态过滤 (Priority: P3)

"关注"标签下展示的动态内容仅来自当前用户已关注的人，保持与关注关系实时一致。

**Why this priority**: 本功能在后端的"关注"筛选逻辑已存在（`filter=following`），此故事确认其与新建的关注数据正确联动，优先级低于前两个核心交互。

**Independent Test**: 关注某用户后切换到"关注"标签，该用户的动态出现在列表中；取消关注后刷新，该用户动态不再显示。

**Acceptance Scenarios**:

1. **Given** 用户关注了某人，**When** 在"关注"标签下刷新动态，**Then** 该用户发布的动态出现在列表中
2. **Given** 用户取消关注了某人，**When** 在"关注"标签下刷新动态，**Then** 该用户发布的动态不再出现

---

### Edge Cases

- 用户尝试关注已被注销或权限受限的账号时，系统给出友好提示而非报错。
- 用户在极短时间内多次点击关注/取消关注按钮时，系统防抖处理，只提交最终状态，避免接口重复请求。
- 关注用户列表在网络请求失败时，显示加载失败提示并提供重试入口。
- 已关注用户数量为 0 时，"关注"标签下的动态列表显示"暂无内容"的空状态提示。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在每条他人动态的作者信息区域提供"关注 / 已关注"切换按钮，当前用户自己的动态不展示该按钮
- **FR-002**: 系统 MUST 在用户点击"关注"后，将关注关系持久化到后端，并即时更新按钮状态为"已关注"
- **FR-003**: 系统 MUST 在用户点击"已关注"后，删除关注关系，并即时更新按钮状态为"关注"
- **FR-004**: 系统 MUST 在"关注"筛选标签激活时，于筛选标签行正下方渲染一行横向可滚动的已关注用户列表（头像 + 姓名）
- **FR-005**: 系统 MUST 在切换至非"关注"标签时，完全隐藏已关注用户列表行，不保留占位空间
- **FR-006**: 系统 MUST 在已关注用户列表为空时，在该行显示空状态文案而非空白区域
- **FR-007**: 系统 MUST 提供获取当前用户已关注用户列表的接口，供"关注标签下用户列表"使用
- **FR-008**: 系统 MUST 对关注/取消关注操作进行防抖处理，避免重复提交
- **FR-009**: 系统 MUST 在关注/取消关注失败时回退按钮状态并展示错误提示
- **FR-010**: 已关注用户在动态卡片中的关注按钮状态 MUST 与关注用户列表的数据保持一致（即时同步）

### Key Entities

- **关注关系（Follow）**: 表示一个用户（关注者）对另一个用户（被关注者）的关注，包含关注者 ID、被关注者 ID、关注时间；关系唯一，不允许重复关注同一人
- **已关注用户列表（FollowedUsers）**: 当前登录用户关注的所有用户的摘要信息集合，包含用户 ID、姓名、头像；用于"关注"标签下的横向展示

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户点击关注/取消关注按钮后，按钮状态切换响应时间在视觉上不超过 300ms（乐观更新）
- **SC-002**: "关注"标签激活时，已关注用户列表在 1 秒内完成加载并展示
- **SC-003**: 关注用户列表与动态卡片上的关注按钮状态保持一致，不出现数据不一致的情况
- **SC-004**: 用户完成"关注某人"全流程操作（发现动态 → 点击关注 → 切换至关注标签查看列表）所需步骤不超过 3 步
- **SC-005**: 空状态和错误状态均有明确的用户可见文案，用户无需联系支持即可自助处理

## Assumptions

- 后端 `/api/posts?filter=following` 接口已存在，本功能通过新增关注关系数据与之联动，无需重建该筛选逻辑
- 当前用户身份通过现有会话机制识别，无需额外鉴权设计
- 关注功能不涉及"互相关注"或"粉丝数"展示，仅实现单向关注与自身关注列表展示
- 已关注用户列表行仅展示头像和姓名，点击用户跳转主页的功能不在本期范围内
- 关注数量无上限限制（可在后续迭代中补充）
