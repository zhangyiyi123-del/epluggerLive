# 功能规格说明：易普圈与企业微信工作台对接

**Feature Branch**: `005-wecom-integration`  
**Created**: 2025-03-12  
**Status**: Draft  
**Input**: 当前易普圈 web 移动端已完成基础功能对接，现需与企业微信对接，员工通过企业微信工作台直接进行身份验证、登录。

## Clarifications

### Session 2025-03-12

- Q: 企微成员与易普圈账号的绑定策略：首次从企微进入时采用哪种方式？ → A: 首次从企微进入时自动创建易普圈账号并绑定该企微成员；可选后续用手机号等方式与已有账号合并。
- Q: 非企微环境（浏览器直接打开）下是否保留手机号登录？ → A: 保留；浏览器直接打开时仍提供手机号+密码/验证码登录，与企微工作台登录并存。
- Q: 是否获取企微敏感信息（如手机号）用于关联或合并账号？ → A: 暂不需要；首期仅用企微 userid/姓名等非敏感信息自动建号与绑定，不申请手机号；后续若做账号合并再单独申请敏感信息。
- Q: 易普圈在企业微信侧的应用类型？ → A: 企业自建应用；仅本企业使用，使用企业 corpid + 自建应用 secret，网页授权与「获取访问用户身份」按自建应用文档实现。
- Q: 非企微环境打开工作台链接时是否先做环境检测再决定登录方式？ → A: 需要；仅当判定为企微内置浏览器时走企微授权免密登录；非企微环境直接展示手机号登录，不发起企微授权。
- **入口**：员工从企业微信工作台打开易普圈应用（H5 或工作台应用入口），在企微环境内完成身份校验与登录，无需在应用内再次输入手机号/密码。

## 用户场景与验收 *(必填)*

### 用户故事 1 - 从企业微信工作台免密登录 (Priority: P1)

员工在企业微信工作台中打开易普圈应用，由企业微信完成身份校验；校验通过后自动完成易普圈登录并进入首页，无需在应用内输入手机号或密码。

**优先级理由**：工作台是主要使用场景，免密登录是企微对接的核心价值。

**独立验收**：从企微工作台打开易普圈，在已配置且身份有效的前提下，用户无需输入账号密码即可进入首页并正常使用各项功能，即视为本故事通过。

**验收场景**：

1. **Given** 员工已加入企业且易普圈已配置在工作台，**When** 在工作台点击易普圈入口，**Then** 跳转至企微授权/登录流程（或直接进入应用若已在企微内完成身份校验）。
2. **Given** 企微身份校验通过，**When** 后端完成用户识别/绑定并签发易普圈登录态，**Then** 前端进入首页或上次访问页，可正常调用现有接口（如首页聚合、打卡、圈子等）。
3. **Given** 用户从企微工作台打开应用且已具备有效登录态，**When** 再次打开或刷新，**Then** 保持已登录状态，无需重新授权。
4. **Given** 企微授权失败或用户非企业成员，**When** 身份校验未通过，**Then** 展示明确提示（如「仅限企业成员使用」或「授权失败，请重试」），不暴露技术堆栈。

---

### 用户故事 2 - 身份与现有账号关联 (Priority: P2)

企业微信成员身份与易普圈已有用户账号正确关联；若为首次通过企微进入，系统按约定策略创建或绑定账号，保证积分、打卡、圈子等数据与同一用户一致。

**优先级理由**：关联错误会导致数据错乱或一人多号，影响激励与统计准确性。

**独立验收**：同一员工无论从企微工作台还是手机号登录进入，看到的个人数据一致；首次从企微进入后，其打卡/积分/动态等均归属该成员，即视为本故事通过。

**验收场景**：

1. **Given** 后端已配置企微成员与易普圈用户的绑定关系，**When** 该成员从工作台登录，**Then** 登录后展示的积分、打卡记录、我的动态等与其账号一致。
2. **Given** 某企微成员首次从工作台打开易普圈，**When** 系统仅凭企微 userid/姓名等非敏感信息自动创建账号并绑定，**Then** 完成首次登录且后续打卡/积分/动态等数据归属该账号；首期不依赖企微手机号，后续可扩展账号合并。
3. **Given** 绑定或创建失败（如企微返回的用户信息不完整、重复等），**When** 后端无法完成关联，**Then** 前端展示明确提示并可引导联系管理员或重试，不暴露技术错误信息。

---

### 边界与异常

- 前端需做运行环境检测（如 UA 或企微 JS-SDK）：仅当判定为企微内置浏览器时走企微授权免密登录；非企微环境（如系统浏览器打开工作台链接）不发起企微授权，直接展示手机号+密码/验证码登录页，与企微登录并存。
- 企微授权超时、用户取消授权、非企业成员访问时，展示明确文案（如「仅限企业成员」或「授权失败，请重试」），不展示错误堆栈。
- 后端需校验 redirect_uri、state 等，防止授权劫持；前端不向用户展示 code 等敏感参数。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须支持从企业微信工作台打开易普圈时，通过企业微信网页授权（或等效能力）完成用户身份校验，并在校验通过后为前端签发易普圈登录态（如 JWT），使员工无需在应用内输入手机号/密码即可使用应用。
- **FR-002**：系统必须在企微成员首次从工作台进入时，仅基于企微 userid/姓名等非敏感信息自动创建易普圈账号并绑定该成员；首期不申请企微手机号等敏感信息；后续若需支持与已有手机号账号合并，再单独申请敏感信息并实现合并逻辑。同一成员在企微登录与手机号登录下数据一致（合并场景在后续迭代）。
- **FR-003**：系统必须在企微授权失败、非企业成员访问或关联失败时，向前端返回或展示明确、用户可理解的提示，不暴露技术错误或堆栈。
- **FR-005**：前端必须做运行环境检测，仅在企微内置浏览器内走企微授权流程；非企微环境直接展示手机号登录，不发起企微授权请求。
- **FR-004**：在非企微环境（如浏览器直接访问）下，必须保留现有手机号+密码/验证码登录能力，与企微工作台登录并存，供非工作台场景使用。

### 关键实体

- **企业微信侧**：企业 ID、应用 ID/AgentId、网页授权 code、成员 userid 及姓名等非敏感信息；首期不申请手机号等敏感信息；授权 redirect_uri 需在企微后台配置为可信域名。
- **易普圈侧**：现有用户表、JWT 登录态；需增加与企微 userid 的绑定字段或映射表。首次从企微进入时仅凭 userid/姓名自动创建用户并写入绑定关系；流程为「企微 code → 换取 userid/姓名 → 用户识别或创建与绑定 → 签发 JWT」。后续若做账号合并再单独申请企微敏感信息。

## 成功标准 *(必填)*

### 可度量结果

- **SC-001**：从企业微信工作台打开易普圈，在已配置且网络正常的前提下，用户在授权通过后 5 秒内进入首页并可持续使用（如查看首页、打卡、圈子）。
- **SC-002**：同一员工通过企微工作台登录与手机号登录所见的个人数据（积分、打卡、动态等）一致。
- **SC-003**：授权失败或非成员访问时，页面展示明确提示且不出现未捕获错误或技术堆栈。

## 假设与依赖

- 易普圈在企业微信侧为企业**自建应用**（仅本企业使用）。企业已在企微管理端创建该自建应用并配置为工作台应用，已配置可信域名与授权回调地址；后端使用企业 corpid + 自建应用 secret 获取 access_token，按自建应用文档完成网页授权与「获取访问用户身份」。
- 企业微信网页授权与「获取访问用户身份」等接口的调用方式与限制以企业微信自建应用官方文档为准（如构造授权链接、code 一次性、redirect_uri 白名单等）。
- 易普圈现有登录态（JWT）与接口鉴权方式不变，企微对接仅增加「通过 code 换取用户身份并签发 JWT」的入口，业务接口仍依赖现有 JWT。
