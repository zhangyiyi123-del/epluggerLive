# Feature Specification: 打卡同步到圈子（「同步到圈子」）

**Feature Branch**: `006-sync-checkin-community`  
**Created**: 2026-03-23  
**Status**: Draft  
**Input**: User description: "针对运动打卡和正向打卡，我想增加一个功能，在按钮上方增加一个，类似同步到圈子中的功能，默认选中，确认打卡后同步发送圈子，名字你帮我命名"

## 命名说明

- **功能在界面上的名称（建议）**：**「同步到圈子」**  
- **含义**：用户在完成运动打卡或正向打卡并确认提交时，除完成打卡外，可选择是否同时在圈子中生成一条动态；勾选表示同步，取消勾选表示仅打卡、不同步到圈子。

## Clarifications

### Session 2026-03-23

- Q: 同步到圈子成功后，用户是否获得与手动发圈一致的积分？ → A: 是；同步成功并生成圈子动态后，用户按**与手动发布圈子动态相同**的规则获得发圈积分；打卡积分仍按既有打卡规则单独核算，二者可同时成立（「也将」表示在打卡积分之外叠加发圈积分），并遵守产品既有积分上限、风控与反作弊规则。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 提交前可选择是否同步到圈子（Priority: P1）

用户在运动打卡或正向打卡流程中，在最终确认/提交按钮的**上方**看到「同步到圈子」选项；该选项**默认处于选中状态**。用户可在提交前随时取消勾选或重新勾选。

**Why this priority**：没有可控的同步开关就无法满足「可选同步」与「默认同步」的核心诉求，也无法独立验收。

**Independent Test**：仅实现该选项的展示、默认状态与可切换，即可在两种打卡页分别完成手测验收。

**Acceptance Scenarios**:

1. **Given** 用户进入运动打卡提交界面（或等价最终确认步骤），**When** 页面展示提交区域，**Then** 在提交按钮上方出现「同步到圈子」选项，且默认已选中。  
2. **Given** 用户进入正向打卡提交界面（或等价最终确认步骤），**When** 页面展示提交区域，**Then** 在提交按钮上方出现「同步到圈子」选项，且默认已选中。  
3. **Given** 选项默认选中，**When** 用户点击取消勾选，**Then** 选项变为未选中且在本次提交前保持用户选择（直至用户再次改动）。  
4. **Given** 选项未选中，**When** 用户再次勾选，**Then** 选项变为选中。

---

### User Story 2 - 确认打卡后按选项同步到圈子（Priority: P1）

用户点击确认并完成打卡后：若「同步到圈子」为选中，则系统在打卡成功的同时，在圈子中生成一条与本次打卡相对应的动态；若为未选中，则**仅**完成打卡，不在圈子中新增动态。

**Why this priority**：这是用户描述的「确认打卡后同步发送圈子」的直接价值交付。

**Independent Test**：在打卡成功的前提下，分别用选中/未选中各提交一次，检查圈子是否仅在有选中时出现对应新动态。

**Acceptance Scenarios**:

1. **Given** 「同步到圈子」为选中且打卡提交通过校验并成功，**When** 用户确认打卡，**Then** 打卡成功，且圈子中出现一条可识别的、与本次打卡关联的动态。  
2. **Given** 「同步到圈子」为未选中且打卡成功，**When** 用户确认打卡，**Then** 打卡成功，且圈子中**不**因本次操作新增动态。  
3. **Given** 用户在同一会话内连续多次打卡，**When** 每次提交前对「同步到圈子」的选择不同，**Then** 每次圈子是否出现新动态与当次选择一致。  
4. **Given** 「同步到圈子」为选中且打卡与同步均成功，**When** 用户确认打卡，**Then** 用户获得与当次操作一致的打卡积分（若有），并**额外**获得与「手动发布等价圈子动态」一致的发圈积分（数额与规则以产品既有发圈积分规则为准）。

---

### User Story 3 - 同步失败时的体验（Priority: P2）

当打卡已成功，但同步到圈子未能完成时，用户应能明确知晓「打卡已成功、圈子未同步」，并知晓可采取的后续动作（例如稍后在圈子手动发布或重试同步——具体文案与是否提供重试以产品为准，但**不得**让用户误以为打卡失败）。

**Why this priority**：避免双重操作带来的误解与重复打卡；降低客服与客诉风险。

**Independent Test**：通过模拟或测试环境制造「打卡成功、同步失败」场景，检查提示与打卡结果一致性。

**Acceptance Scenarios**:

1. **Given** 打卡已成功，**When** 同步到圈子失败，**Then** 用户看到明确区分「打卡成功」与「未能同步到圈子」的提示信息。  
2. **Given** 同步到圈子失败，**When** 用户查看打卡记录/结果，**Then** 打卡结果仍显示为成功（与业务规则一致），不因同步失败而被撤销或显示为失败。

---

### Edge Cases

- 未登录或会话失效时提交：应遵循产品既有登录/会话处理；不要求在规格层规定技术实现。  
- 用户无圈子发布权限或被策略禁止发布时：打卡是否仍允许成功、同步选项是否禁用或自动取消——**默认假设**：打卡仍可按原规则成功；若无法同步，应给出清晰说明（与 User Story 3 一致）。  
- 用户在极短时间内重复点击提交：应避免重复产生多条相同圈子动态（与产品既有防重复提交策略对齐）。  
- 正向打卡与运动打卡的文案或摘要字段不同：同步到圈子的动态内容应分别基于两类打卡中用户可见的关键信息生成，且不含敏感字段的误暴露（遵循既有隐私与可见范围规则）。  
- **积分**：若用户未勾选同步，则**不得**因当次打卡发放发圈积分；若勾选且同步成功，则发放发圈积分且**不得**因「来源为打卡同步」而低于同等条件下手动发圈应得积分（除非触发风控拒发或上限截断，此时须与手动发圈的提示与记录方式一致）。若动态已发布但积分侧异常未入账，应有与产品一致的补救或说明路径（可与客服/积分流水核对流程对齐，实现阶段细化）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：系统 MUST 在运动打卡与正向打卡的**最终确认/提交区域**，于提交按钮**上方**提供「同步到圈子」选项。  
- **FR-002**：该选项 MUST **默认选中**。  
- **FR-003**：用户 MUST 能在提交前自由切换选中/未选中状态。  
- **FR-004**：当用户确认打卡且打卡成功时，若选项为选中，系统 MUST 在圈子中创建一条与本次打卡相对应的动态。  
- **FR-005**：当用户确认打卡且打卡成功时，若选项为未选中，系统 MUST **不**因本次打卡在圈子中创建动态。  
- **FR-006**：若打卡成功但同步到圈子失败，系统 MUST 向用户明确提示打卡成功与同步失败的分层信息，且不得将打卡结果表现为失败（除非业务上打卡本身失败）。  
- **FR-007**：同步到圈子所产生的动态，其**可见范围** MUST 与产品对「从打卡生成圈子内容」的默认规则一致；本规格默认与「用户日常发布圈子动态的默认可见范围」一致，若产品已有单独规则则以既有规则为准。  
- **FR-008**：同步到圈子的动态内容 MUST 能让用户识别来源于哪一次打卡（运动或正向），且不泄露仅应在打卡域展示的限制性信息。实现上正文由服务端按约定模板生成，并可在文末使用话题标签（如 `#来自打卡同步`）辅助识别；**字段映射与例句**见 `contracts/README.md` §7，仍须满足本条的隐私与可识别性要求。  
- **FR-009**：当打卡成功且「同步到圈子」选中且圈子动态创建成功时，系统 MUST 按**与手动发布该条圈子动态等效**的规则向用户计入发圈积分；打卡积分仍按既有打卡规则单独处理。未勾选同步时 MUST 不因当次打卡计入发圈积分。积分发放 MUST 遵守产品既有的每日/单次上限、风控与反作弊规则（与手动发圈一致）。

### Key Entities *(include if feature involves data)*

- **打卡记录**：一次运动打卡或正向打卡的提交结果；可与圈子动态建立业务上的对应关系（用户可见层面表现为「来自打卡」）。  
- **圈子动态**：展示在圈子中的内容条目；在勾选同步时由打卡成功事件触发创建。  
- **同步选项状态**：用户在当次提交前的选择（默认选中）；用于决定是否在打卡成功后创建圈子动态。

### Assumptions

- 「确认打卡」指用户完成校验后触发的最终提交动作，与现有打卡成功定义一致。  
- 圈子动态的正文句式与话题标签以实现文档为准（`contracts/README.md` §7、`data-model.md` §5.1），须满足 FR-008 的可识别性与隐私要求；列表页展示样式仍遵循产品既有圈子规范。  
- 若未来需要「记住用户上次是否勾选」，可作为增强项；本规格仅要求单次流程内选择与提交行为一致。  
- 发圈积分的具体数额、阶梯与上限以产品现行规则为准；本规格只要求「同步生成动态」与「手动发圈」在积分规则上**等效**，不因入口为打卡而克扣。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：在可用性测试中，≥90% 的受试用户能在首次进入任一类打卡提交页时，在 10 秒内发现「同步到圈子」选项并正确说出默认状态为「已勾选」。  
- **SC-002**：在结构化验收中，对运动打卡与正向打卡各执行不少于 5 次「选中同步」提交，圈子侧均出现与当次打卡一一对应的新动态，误判率 0。  
- **SC-003**：在同一验收中，各执行不少于 5 次「取消同步」提交，圈子侧均不因当次提交新增动态，误判率 0。  
- **SC-004**：在模拟同步失败场景中，100% 的用例下用户能区分「打卡已成功」与「未同步到圈子」，且无人将同步失败误解为打卡失败（以任务后问卷或访谈记录为准）。  
- **SC-005**：在结构化验收中，对「选中同步且动态创建成功」的样本（运动与正向各不少于 3 次），用户在积分侧获得的发圈积分与「同等内容假设下手动发圈」的预期一致（在相同规则与未触发风控的前提下，差异为 0）；未勾选同步的样本不得出现因当次提交产生的发圈积分记录。
