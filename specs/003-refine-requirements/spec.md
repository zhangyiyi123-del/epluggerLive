## Clarifications

### Session 2025-03-09（沿用已决结论）

- 用户角色：仅员工一种角色；所有已登录用户权限相同，无管理员/运营角色。
- 消息通知：仅站内消息；用户只在应用内「我的消息」等入口查看，不做推送。
- 历史记录与积分等数据：与账号同生命周期保留；账户注销与数据清理列为后续迭代。
- 同一用户多端/多标签同时编辑同一动态：后写覆盖，以最后提交为准，不提供冲突提示。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 登录与访问控制 (Priority: P1)

员工通过多种方式登录企业正向激励应用；未登录时只能访问登录页，登录后可持续使用且再次打开应用时保持登录状态。

**Why this priority**: 无登录则无法使用任何业务功能，是后续所有模块的前置条件。

**Independent Test**: 完成任一种登录方式并关闭后再次打开仍处于已登录状态；未登录时访问任意业务页均被重定向到登录页，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户未登录，**When** 访问任意非登录页，**Then** 被重定向至登录页。
2. **Given** 用户在登录页，**When** 使用密码登录（手机号 + 密码）且格式与校验通过，**Then** 登录成功并进入首页。
3. **Given** 用户在登录页，**When** 使用验证码登录（手机号 + 6 位验证码）且校验通过，**Then** 登录成功并进入首页。
4. **Given** 用户已登录，**When** 关闭应用后再次打开，**Then** 仍为已登录状态，无需重新登录。
5. **Given** 用户在登录页，**When** 手机号格式错误或密码/验证码为空提交，**Then** 显示明确错误提示且不发起登录。

---

### User Story 2 - 运动打卡 (Priority: P2)

员工记录运动类型、时长/距离/卡路里、强度及佐证图片，查看每日与周期目标进度及历史记录。

**Why this priority**: 运动打卡是核心激励行为之一，直接产生积分与排行榜数据。

**Independent Test**: 完成一次完整运动打卡并能在历史记录中看到该条记录，且能看到当日/周期目标进度，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 选择运动类型并填写时长/距离/卡路里与强度，**Then** 可提交打卡并看到成功反馈。
2. **Given** 用户在打卡页，**When** 上传佐证图片（最多 3 张），**Then** 图片与打卡一起保存并可查看。
3. **Given** 用户在打卡相关页，**When** 查看当日或本周目标，**Then** 能看到完成进度展示。
4. **Given** 用户已提交过运动打卡，**When** 进入运动历史记录页，**Then** 能按时间查看自己的历史打卡列表。

---

### User Story 3 - 正向行为打卡 (Priority: P3)

员工选择正向行为分类、填写描述、可选 @同事 与佐证，查看积分奖励预览及历史记录。

**Why this priority**: 与运动打卡并列的核心激励模块，支撑企业文化与正向行为可见化。

**Independent Test**: 完成一次正向行为打卡并在历史记录中可见，且能看到积分奖励预览，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 选择行为分类、填写描述并可选 @同事 与佐证，**Then** 可提交并看到成功反馈。
2. **Given** 用户在正向打卡页，**When** 填写完必填项，**Then** 能看到本次行为对应的积分奖励预览。
3. **Given** 用户已提交过正向打卡，**When** 进入正向记录历史页，**Then** 能按时间查看自己的历史记录列表。

---

### User Story 4 - 圈子社交 (Priority: P4)

员工浏览动态列表（筛选、分页）、发布动态（富文本、多图、话题、@、可见范围）、在详情页点赞/收藏/评论与二级回复，并管理「我的动态」。

**Why this priority**: 社交与内容沉淀，增强粘性与正向行为传播。

**Independent Test**: 能浏览动态列表、发布一条动态、在详情页完成点赞与一条评论，并在「我的动态」中看到刚发布的动态，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 进入圈子页并切换筛选（如最新/热门/本部门/关注），**Then** 列表按所选维度展示且支持分页加载。
2. **Given** 用户在发布页，**When** 输入正文、上传图片（可选）、选择话题与@同事、设置可见范围并发布，**Then** 发布成功且新动态出现在列表或我的动态中。
3. **Given** 用户在动态详情页，**When** 进行点赞、收藏或发表评论/回复，**Then** 操作成功且数量/列表即时更新。
4. **Given** 用户进入「我的动态」，**When** 查看列表，**Then** 仅展示当前用户发布的动态，并可进行编辑/删除等管理操作。

---

### User Story 5 - 积分体系 (Priority: P5)

员工查看积分统计（累计/已用/可用）、等级进度、勋章墙（已获得/待解锁）、积分商城兑换、排行榜（积分/运动/正向）及消息通知（点赞/评论/@）。

**Why this priority**: 积分与勋章是激励闭环的核心展示，影响持续使用意愿。

**Independent Test**: 能查看当前积分与等级、勋章墙、排行榜，并能在消息入口看到互动通知，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 进入积分相关页，**Then** 能看到累计/已使用/可用积分及当前等级与进度。
2. **Given** 用户查看勋章墙，**When** 浏览列表，**Then** 能区分已获得与待解锁勋章。
3. **Given** 用户在积分商城，**When** 选择商品并确认兑换，**Then** 扣除对应积分并完成兑换流程（含确认步骤）。
4. **Given** 用户查看排行榜，**When** 切换榜单类型（积分/运动/正向）与时间范围，**Then** 列表随选择更新。
5. **Given** 用户进入消息/通知，**When** 查看列表，**Then** 能看到点赞、评论、@提及等互动通知并可跳转到对应动态。

---

### User Story 6 - 个人中心 (Priority: P6)

员工在个人中心查看连续打卡天数、累计积分、勋章数，使用「我的动态」「我的消息」「通知设置」「帮助与反馈」入口，使用深色模式开关（UI）并退出登录。

**Why this priority**: 统一入口与账号管理，保证体验完整与可退出。

**Independent Test**: 能查看个人统计数据、进入各菜单入口、切换深色模式展示、退出后再次进入需重新登录，即视为本故事通过。

**Acceptance Scenarios**:

1. **Given** 用户已登录，**When** 进入个人中心，**Then** 能看到连续打卡天数、累计积分、获得勋章数等统计。
2. **Given** 用户在个人中心，**When** 点击「我的动态」「我的消息」「通知设置」「帮助与反馈」，**Then** 进入对应功能或页面。
3. **Given** 用户在个人中心，**When** 切换深色模式开关，**Then** 界面以深色模式展示（或按设计展示开关状态）。
4. **Given** 用户已登录，**When** 点击退出登录，**Then** 登录态清除，再次访问需重新登录。

---

### Edge Cases

- 同一用户在不同设备或标签页同时编辑同一动态时，以最后提交为准，不提供冲突提示。
- 网络异常或超时时，提交操作（登录、打卡、发布、兑换）应有明确失败提示并可重试。
- 列表/分页在无数据时展示空状态说明，不显示错误堆栈。
- 未登录或登录过期时，所有需登录的接口/页面统一重定向至登录页并提示。
- 积分不足时兑换流程应阻止提交并提示可用积分不足。
- 图片/佐证上传数量与格式需符合约定（如佐证最多 3 张），超出或非法格式时给出明确提示。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须支持三种登录方式：密码登录（手机号+密码）、验证码登录（手机号+6位验证码）、企业 SSO 登录入口。
- **FR-002**: 系统必须在客户端持久化登录态；未登录时除登录页外所有页面访问必须重定向至登录页。
- **FR-003**: 系统必须对登录表单进行校验：手机号格式、密码/验证码非空，并在不通过时给出错误提示。
- **FR-004**: 系统必须支持运动打卡：多类型运动选择、打卡数据录入（时长/距离/卡路里）、强度选择、佐证图片上传（最多 3 张）、目标进度展示、历史记录查看。
- **FR-005**: 系统必须支持正向行为打卡：行为分类选择、同事 @提及、行为描述、佐证上传、积分奖励预览、历史记录查看。
- **FR-006**: 系统必须支持圈子动态列表：按维度筛选（如最新/热门/本部门/关注）、分页加载。
- **FR-007**: 系统必须支持发布动态：富文本、多图、话题标签、@同事、可见范围设置。
- **FR-008**: 系统必须支持动态详情：点赞、收藏、评论及二级回复。
- **FR-009**: 系统必须支持「我的动态」：仅展示当前用户发布的动态，并支持编辑/删除等管理操作。
- **FR-010**: 系统必须支持积分体系：积分统计（累计/已用/可用）、等级进度展示、勋章墙（已获得/待解锁）、积分商城（商品展示与兑换）、排行榜（积分/运动/正向）、消息通知（点赞/评论/@）；消息通知仅在应用内入口展示，不提供推送通知。
- **FR-011**: 系统必须在个人中心展示：连续打卡天数、累计积分、勋章数；并提供我的动态、我的消息、通知设置、帮助与反馈入口。
- **FR-012**: 系统必须支持个人中心内深色模式切换的 UI 展示，以及退出登录并清除登录态。
- **FR-013**: 系统仅支持单一用户角色（员工）；所有已登录用户具备相同权限，不提供管理员或运营角色。

### Key Entities

- **用户**：身份标识、姓名、头像、部门、岗位等基础信息；与登录态、打卡记录、动态、积分、勋章、消息关联。
- **打卡记录（运动）**：运动类型、时长/距离/卡路里、强度、佐证图片、所属用户、时间等。
- **打卡记录（正向）**：行为分类、描述、@提及、佐证、积分奖励、所属用户、时间等。
- **动态**：作者、正文内容、图片/视频、话题、@提及、可见范围、点赞/评论数、是否已点赞/收藏、发布时间等。
- **评论**：所属动态、作者、内容、父评论（二级回复）、点赞数、时间等。
- **积分与等级**：用户累计/已用/可用积分、当前等级、等级进度等。
- **勋章**：类型、获得时间、与用户的获得关系（已获得/待解锁）。
- **消息/通知**：类型（点赞/评论/@）、关联动态与操作者、是否已读、时间等。
- **积分商品与订单**：商品信息、所需积分、兑换订单等。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户从打开登录页到完成登录并进入首页，在信息正确的前提下可在 2 分钟内完成。
- **SC-002**: 用户完成单次运动或正向打卡（含选类型、填数据、选佐证）可在 3 分钟内完成并看到成功反馈。
- **SC-003**: 用户从进入圈子到发布一条带文字与图片的动态，可在 5 分钟内完成并能在列表中看到该条动态。
- **SC-004**: 用户在动态详情页进行点赞、评论或二级回复操作后，界面在 3 秒内反映数量或列表变化（在正常网络条件下）。
- **SC-005**: 用户可在个人中心与积分相关页稳定查看自己的积分、等级与勋章，数据与行为结果一致。
- **SC-006**: 用户退出登录后再次打开应用时，必须重新登录才能访问业务功能，无越权访问。

## Assumptions & Dependencies

- 用户可见的打卡、动态、积分、消息等数据随账号存在而保留，本规格不做单独数据保留期约定；账户注销与数据清理不在本期范围，列为后续迭代。
- 企业 SSO 的认证协议与端点由企业侧提供，本规格仅约定「提供 SSO 入口并支持跳转与回调」。
- 手机号格式、验证码长度与有效期、密码策略遵循企业或产品统一规范。
- 可见范围（如全公司、本部门）与组织架构数据由后端或现有系统提供。
- 积分规则、等级与勋章类型、商城商品与兑换规则由业务/后端定义，本规格不规定具体数值与策略。
- 深色模式在本规格中仅要求 UI 具备开关与展示能力，是否持久化与全局生效可在实现阶段约定。
