## 一、产品概述与目标

EPlugger 为企业正向激励社区移动端 Web 应用，围绕**运动打卡、正向行为打卡、圈子社交、积分体系**四大核心模块，通过积分、勋章、排行榜等机制，帮助员工养成运动习惯并营造正向企业文化。

**目标用户**：企业员工（单一角色，无管理员/运营角色；所有已登录用户权限一致）。

## 三、用户场景与验收标准

### 用户故事 1 - 登录与访问控制（P1）

员工通过多种方式登录；未登录仅能访问登录页，登录后可持续使用，再次打开应用保持登录状态。

**独立验收**：任一种登录方式成功且关闭后再次打开仍为已登录；未登录访问任意业务页均重定向到登录页即通过。

**验收场景**：

1. **Given** 未登录 **When** 访问任意非登录页 **Then** 重定向至登录页。
2. **Given** 在登录页 **When** 使用手机号+密码且校验通过 **Then** 登录成功并进入首页。
3. **Given** 在登录页 **When** 使用手机号+6 位验证码且校验通过 **Then** 登录成功并进入首页。
4. **Given** 已登录 **When** 关闭应用后再次打开 **Then** 仍为已登录状态。
5. **Given** 在登录页 **When** 手机号格式错误或密码/验证码为空提交 **Then** 显示错误提示且不发起登录。

**与前端对应**：`LoginPage`、`App.tsx` 中 `isLoggedIn` 与 `localStorage('ep_logged_in')`、未登录时 `<Navigate to="/login" />`。

---

### 用户故事 2 - 运动打卡（P2）

员工记录运动类型、时长/距离/卡路里、强度及佐证图片，查看每日与周期目标进度及历史记录。

**独立验收**：完成一次完整运动打卡，历史记录中可见该条，且能看到当日/周期目标进度即通过。

**验收场景**：

1. **Given** 已登录 **When** 选择运动类型并填写时长/距离/卡路里与强度 **Then** 可提交并看到成功反馈。
2. **Given** 在打卡页 **When** 上传佐证（最多 3 张）**Then** 与打卡一起保存并可查看。
3. **Given** 在打卡相关页 **When** 查看当日或本周目标 **Then** 能看到完成进度。
4. **Given** 已提交过运动打卡 **When** 进入运动历史记录页 **Then** 能按时间查看历史列表。

**与前端对应**：`CheckInPage`、`CheckInForm`、`SportTypeSelector`、`AttachmentUpload`、`DailyGoalCard`、`CycleGoalProgress`、`ExerciseRecordsPage`，路由 `/checkin`、`/checkin/exercise-records`。

---

### 用户故事 3 - 正向行为打卡（P3）

员工选择正向行为分类、填写描述、可选 @同事 与佐证，查看积分奖励预览及历史记录。

**独立验收**：完成一次正向打卡且历史记录中可见，并看到积分奖励预览即通过。

**验收场景**：

1. **Given** 已登录 **When** 选择行为分类、填写描述并可选 @同事与佐证 **Then** 可提交并看到成功反馈。
2. **Given** 在正向打卡页 **When** 填写完必填项 **Then** 能看到本次积分奖励预览。
3. **Given** 已提交过正向打卡 **When** 进入正向记录历史页 **Then** 能按时间查看历史列表。

**与前端对应**：`PositiveCheckInPage`、`PositiveCheckInForm`、`CategoryTagSelector`、`ColleagueSelector`、`PositiveEvidenceUpload`、`PositiveRecordsPage`，路由 `/checkin/positive`、`/checkin/positive-records`。

---

### 用户故事 4 - 圈子社交（P4）

员工浏览动态列表（筛选、分页）、发布动态（富文本、多图、话题、@、可见范围）、在详情页点赞/收藏/评论与二级回复，并管理「我的动态」。

**独立验收**：能浏览动态列表、发布一条动态、在详情页完成点赞与一条评论，且在「我的动态」中看到刚发布的动态即通过。

**验收场景**：

1. **Given** 已登录 **When** 进入圈子并切换筛选（最新/热门/本部门/关注）**Then** 列表按所选维度展示且支持分页。
2. **Given** 在发布页 **When** 输入正文、上传图片、选话题与@、设置可见范围并发布 **Then** 发布成功且新动态出现在列表或我的动态中。
3. **Given** 在动态详情页 **When** 点赞、收藏或发表评论/回复 **Then** 操作成功且数量/列表即时更新。
4. **Given** 进入「我的动态」**When** 查看列表 **Then** 仅展示当前用户发布的动态，并可编辑/删除。

**与前端对应**：`CommunityPage`、`PostCard`、`PostComposer`、`PublishPage`、`PostDetailPage`、`CommentList`/`CommentCard`、`MyPostsPage`，路由 `/community`、`/community/:postId`、`/publish`、`/profile/posts`。

---

### 用户故事 5 - 积分体系（P5）

员工查看积分统计、等级进度、勋章墙、积分商城兑换、排行榜及消息通知（点赞/评论/@）。

**独立验收**：能查看当前积分与等级、勋章墙、排行榜，并在消息入口看到互动通知即通过。

**验收场景**：

1. **Given** 已登录 **When** 进入积分相关页 **Then** 能看到累计/已使用/可用积分及等级与进度。
2. **Given** 查看勋章墙 **When** 浏览列表 **Then** 能区分已获得与待解锁。
3. **Given** 在积分商城 **When** 选择商品并确认兑换 **Then** 扣除积分并完成兑换（含确认步骤）。
4. **Given** 查看排行榜 **When** 切换榜单类型（积分/运动/正向）与时间范围 **Then** 列表随选择更新。
5. **Given** 进入消息/通知 **When** 查看列表 **Then** 能看到点赞、评论、@ 等互动通知并可跳转对应动态。

**与前端对应**：`LeaderboardPage`、`LevelProgress`、`PointsCenter`、`MedalWall`、`PointsMallPage`、`ProductCard`、`ExchangeModal`、`MyMessagesPage`，路由 `/leaderboard`、`/profile/messages`（及积分商城在排行榜内入口）。

---

### 用户故事 6 - 个人中心（P6）

员工在个人中心查看连续打卡天数、累计积分、勋章数，使用「我的动态」「我的消息」「通知设置」「帮助与反馈」，使用深色模式开关（UI）并退出登录。

**独立验收**：能查看个人统计、进入各菜单入口、切换深色模式展示、退出后再次进入需重新登录即通过。

**验收场景**：

1. **Given** 已登录 **When** 进入个人中心 **Then** 能看到连续打卡天数、累计积分、获得勋章数。
2. **Given** 在个人中心 **When** 点击我的动态/我的消息/通知设置/帮助与反馈 **Then** 进入对应功能或页面。
3. **Given** 在个人中心 **When** 切换深色模式开关 **Then** 界面以深色模式展示（或开关状态符合设计）。
4. **Given** 已登录 **When** 点击退出登录 **Then** 登录态清除，再次访问需重新登录。

**与前端对应**：`ProfilePage`、底部导航「我的」、`/profile`；我的动态/我的消息对应 `/profile/posts`、`/profile/messages`；通知设置、帮助与反馈为菜单入口（具体页待实现）。

---

### 边界与异常

- 多端/多标签同时编辑同一动态：后写覆盖，不提供冲突提示。
- 网络异常/超时：登录、打卡、发布、兑换等提交操作需明确失败提示并可重试。
- 列表/分页无数据：展示空状态，不展示错误堆栈。
- 未登录或登录过期：需登录的接口/页面统一重定向登录页并提示。
- 积分不足：兑换流程阻止提交并提示可用积分不足。
- 佐证/图片：数量与格式符合约定（如佐证最多 3 张），超出或非法格式时明确提示。

## 四、功能需求（FR）

| 编号 | 需求描述 |
|------|----------|
| FR-001 | 支持三种登录方式：密码登录（手机号+密码）、验证码登录（手机号+6 位验证码）、企业 SSO 登录入口。 |
| FR-002 | 在客户端持久化登录态；未登录时除登录页外所有页面重定向至登录页。 |
| FR-003 | 登录表单校验：手机号格式、密码/验证码非空，不通过时给出错误提示。 |
| FR-004 | 运动打卡：多类型运动、时长/距离/卡路里、强度、佐证图最多 3 张、目标进度展示、历史记录查看。 |
| FR-005 | 正向行为打卡：行为分类、同事 @、行为描述、佐证上传、积分奖励预览、历史记录查看。 |
| FR-006 | 圈子动态列表：按维度筛选（最新/热门/本部门/关注）、分页加载。 |
| FR-007 | 发布动态：富文本、多图、话题、@同事、可见范围。 |
| FR-008 | 动态详情：点赞、收藏、评论及二级回复。 |
| FR-009 | 我的动态：仅展示当前用户发布的动态，支持编辑/删除等管理。 |
| FR-010 | 积分体系：积分统计（累计/已用/可用）、等级进度、勋章墙（已获得/待解锁）、积分商城（展示与兑换）、排行榜（积分/运动/正向）、消息通知（点赞/评论/@）；通知仅站内展示，不做推送。 |
| FR-011 | 个人中心：展示连续打卡天数、累计积分、勋章数；提供我的动态、我的消息、通知设置、帮助与反馈入口。 |
| FR-012 | 个人中心支持深色模式切换的 UI 展示，以及退出登录并清除登录态。 |
| FR-013 | 仅支持单一用户角色（员工）；所有已登录用户权限相同，无管理员/运营角色。 |

## 五、关键实体与数据（与前端类型对应）

- **用户**：id、姓名、头像、部门、岗位等；与登录态、打卡、动态、积分、勋章、消息关联。对应 `types/community.ts` 等中的 `User`。
- **运动打卡记录**：运动类型、时长/距离/卡路里、强度、佐证图片、用户、时间。对应 `types/checkIn.ts`。
- **正向打卡记录**：行为分类、描述、@提及、佐证、积分奖励、用户、时间。对应 `types/positive.ts`。
- **动态**：作者、正文、图片/视频、话题、@、可见范围、点赞/评论数、是否已点赞/收藏、发布时间等。对应 `Post`、`PostContent`、`Visibility`。
- **评论**：所属动态、作者、内容、父评论（二级回复）、点赞数、时间。对应 `Comment`。
- **积分与等级**：累计/已用/可用积分、当前等级、等级进度。对应 `UserPoints`、`LevelType`。
- **勋章**：类型、获得时间、与用户的获得关系。对应 `Medal`、`MedalType`。
- **消息/通知**：类型（点赞/评论/@）、关联动态与操作者、已读、时间。对应「我的消息」列表数据。
- **积分商品与订单**：商品信息、所需积分、兑换订单。对应积分商城与 `OrderList` 等。

## 六、成功标准（可度量）

| 编号 | 标准 |
|------|------|
| SC-001 | 用户在信息正确前提下，从打开登录页到进入首页可在 2 分钟内完成。 |
| SC-002 | 单次运动或正向打卡（含选类型、填数据、选佐证）可在 3 分钟内完成并看到成功反馈。 |
| SC-003 | 从进入圈子到发布一条带文字与图片的动态，可在 5 分钟内完成并能在列表中看到该条动态。 |
| SC-004 | 在动态详情页点赞、评论或二级回复后，正常网络下 3 秒内界面反映数量或列表变化。 |
| SC-005 | 在个人中心与积分相关页稳定查看积分、等级与勋章，数据与行为结果一致。 |
| SC-006 | 退出登录后再次打开应用需重新登录才能使用业务功能，无越权访问。 |

## 七、前端实现与需求追溯（简要）

便于开发与测试对照，需求与前端实现的对应关系如下（细节见 `frontend/PROJECT_DOCS.md`）：

- **路由与页面**：登录 `/login` → `LoginPage`；首页 `/` → `HomePage`；打卡 `/checkin`、`/checkin/exercise-records`、`/checkin/positive`、`/checkin/positive-records`；圈子 `/community`、`/community/:postId`、`/publish`；排行与积分 `/leaderboard`；个人中心 `/profile`、`/profile/posts`、`/profile/messages`。
- **登录与守卫**：`App.tsx` 中 `isLoggedIn`、`localStorage('ep_logged_in')`、未登录时 `<Navigate to="/login" />`。
- **底部导航**：首页、打卡、圈子、排行、我的；在 `/community/:postId`、`/publish`、`/profile/posts`、`/profile/messages`、`/login` 隐藏。
- **数据**：当前为 Mock（`MOCK_` 前缀），后续对接 API 替换；类型定义见 `src/types/`（`checkIn.ts`、`community.ts`、`points.ts`、`positive.ts`）。

## 八、假设与依赖

- 打卡、动态、积分、消息等数据随账号存在而保留，本期不约定单独保留期；账户注销与数据清理为后续迭代。
- 企业 SSO 的协议与端点由企业侧提供，本需求仅约定提供 SSO 入口并支持跳转与回调。
- 手机号格式、验证码长度与有效期、密码策略遵循企业或产品规范。
- 可见范围（全公司、本部门等）与组织架构由后端或现有系统提供。
- 积分规则、等级与勋章类型、商城商品与兑换规则由业务/后端定义。
- 深色模式本期仅要求 UI 具备开关与展示能力，持久化与全局策略可在实现阶段约定。
