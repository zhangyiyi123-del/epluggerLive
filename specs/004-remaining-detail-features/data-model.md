# Phase 1: Data Model — 004 无新表，聚合与查询扩展

**Feature**: 004-remaining-detail-features | **Date**: 2025-03-12

**持久化目标**：与 003 一致，**MySQL**；本期**不新增表**，仅扩展查询与聚合逻辑。

## 与 003 实体关系（复用）

| 实体 | 004 相关用法 |
|------|----------------|
| 用户 (User) | 首页个人统计、@ 提及对象、检索作者 |
| 运动打卡记录 (CheckInRecord) | 首页最近 3 条、今日/本周进度、月度汇总 |
| 正向打卡记录 (PositiveRecord) | 首页最近 3 条（与运动合并展示）、@ 提及 |
| 动态 (Post) | 首页热门、圈子列表筛选+检索、@ 提及 |
| 评论 (Comment) | 动态/评论中 @ 展示（本期不新增评论 @） |
| 消息/通知 (Notification) | @ 类型通知、跳转目标 |

无新实体；@ 关系、可见范围、话题等沿用 003 设计（关联表或 JSON 字段）。

## 本期扩展的查询与聚合

### 1. 首页聚合（无新表）

- **今日/本周打卡进度**：基于 CheckInRecord / PositiveRecord 按当前用户、今日/本周时间范围聚合计数或目标完成率；目标值可来自配置或固定规则。
- **个人统计**：积分、连续打卡天数、本周排名— 来自 003 已有积分与排行榜逻辑。
- **最近 3 条打卡记录**：运动与正向打卡按时间排序取最近 3 条（可合并展示或分类型）。
- **热门动态**：基于 Post 的点赞、评论、时间等规则排序取前 N 条；只读查询，无需新表。

### 2. 圈子列表筛选 + 检索（无新表）

- **筛选**：沿用 003 的 `filter`（如 latest / hot / department / following），对应 Post 查询条件与排序。
- **检索**：新增 `keyword`；匹配 Post 正文或作者名称（或扩展至评论，由后端约定）；与 filter 组合为同一查询，分页同 003。

### 3. @ 提及与通知（无新表）

- **写入**：创建 Post 或 PositiveRecord 时写入 mention 关联；同时创建 Notification 类型为 @，relatedPostId / relatedUserId 等指向被提及动态或打卡。
- **展示**：Post/Comment 的 mention 列表与 003 一致；评论中本期仅展示，不新建 @。

### 4. 运动月度统计（无新表）

- **汇总**：按用户 + 月份（yyyy-MM）对 CheckInRecord 聚合：次数、总时长、总距离、总卡路里等；与历史记录明细口径一致（如不包含已删除记录、时区由后端统一）。

## 校验与一致性

- 首页「最近 3 条」严格 3 条；热门条数由后端或产品配置。
- 月度汇总与 GET /api/checkin/exercise/records 该月明细可对照一致。
- 筛选+检索组合结果与单独筛选/单独检索在边界情况下无重复或遗漏（分页参数一致）。

## 与前端类型对齐

- 首页：新类型或扩展 003 的 UserPoints / CheckInRecord 等，与 `frontend/src/types/` 或新 `api/home.ts` 对齐。
- 圈子：`getPosts(filter, keyword?, page, size)` 响应与现有 Post 列表类型一致。
- 月度统计：新 DTO（如 monthlySummary）与打卡页现有展示结构对齐。
